## 3. Sistem replace dokumen — ada riwayat catatan revisi, tapi tidak duplikasi dokumen

> *"coba sistem replace. ada riwayat catatan revisi tapi tidak duplikasi dokumen. jadi semisal surat pernah revisi, langsung menggantikan surat lama agar penyimpanan tidak penuh"*

**Status: ✅ Sudah diimplementasikan ke kode dan divalidasi.**

### Bagian yang sudah terpenuhi: "tidak duplikasi dokumen"

✅ **Sudah benar.** Di `DocumentService::resubmit()`, saat pengaju submit ulang dokumen yang di-reject:

```php
// Hapus lampiran lama dan simpan file baru
$document->loadMissing('attachments');
foreach ($document->attachments as $old) {
    Storage::delete($old->file_path);
    $old->delete();
}

$path = $newFile->store('documents');
$this->documents->createAttachment([...]);
```

File fisik lama **dihapus dari storage** dan record `document_attachments` lama **dihapus dari database**, baru file baru disimpan. Tidak ada duplikasi/penumpukan file untuk dokumen yang sama. Bagian ini **tidak perlu diubah**.

### Bagian yang belum terpenuhi: "ada riwayat catatan revisi"

❌ **Belum ada.** Di `WorkflowApprovalEngine::resubmitDocument()` baris 248-254:

```php
// Reset approval yang ditolak menjadi PENDING kembali
$rejectedApproval->update([
    'status'      => ApprovalStatus::PENDING,
    'notes'       => null,        // ← catatan reject DIHAPUS, bukan disimpan sebagai riwayat
    'approved_at' => null,
    'approved_by' => $document->created_by,
]);
```

Catatan alasan reject (`notes` di `document_approvals`) **di-set `null`** saat resubmit — bukan disimpan sebagai histori. Begitu pengaju submit ulang, catatan reject sebelumnya **hilang permanen**. Tidak ada cara untuk pengaju atau approver berikutnya melihat "kenapa surat ini pernah ditolak" setelah resubmit.

**Dampak ke poin evaluasi lain:** ini adalah akar masalah yang sama untuk poin evaluasi #6 (riwayat catatan revisi), #7 (pembeda nama/notifikasi surat resubmit), dan #8 (tampilkan catatan reject di status dokumen). Poin ini (#4) dan #6/#8 (tampilan) sudah ditangani lewat implementasi di bawah; #7 (pembeda nama/notifikasi) masih terpisah, belum dikerjakan.

### Pendekatan yang Dipilih & Diimplementasikan

Dari 2 opsi yang diajukan, dipilih: **jangan timpa approval yang reject — biarkan sebagai histori, buat approval baru di step_order yang sama untuk siklus resubmit berikutnya.** Tidak perlu tabel baru, hanya 1 kolom tambahan (`created_at`) untuk menjaga urutan kronologis saat ada lebih dari satu baris approval di step_order yang sama.

## 5. Apabila surat pernah kereject, terdapat riwayat catatan revisi sebelumnya (catatan tidak hilang saat dikembalikan)

**Status: ✅ Sudah terpenuhi — hasil samping dari implementasi poin 3, tidak ada kode tambahan yang diubah di sini.**

### Validasi Terhadap Kode Saat Ini

Dicek ulang `WorkflowApprovalEngine.php`:

- `rejectByApprovalId()` baris 191-196 — saat approver reject, `notes` diisi dari input approver dan disimpan ke `DocumentApproval` (status `REJECTED`). Ini tidak berubah dari sebelumnya.
- `resubmitDocument()` (hasil perubahan poin 3) — baris yang dulu melakukan `$rejectedApproval->update(['notes' => null, ...])` **sudah dihapus**. Approval yang reject sekarang **dibiarkan apa adanya** (status REJECTED, notes tetap tersimpan), dan approval baru dibuat terpisah untuk siklus berikutnya.

### Konfirmasi: "Tidak Hilang Saat Dikembalikan"

Karena baris yang dulu menghapus `notes` (`'notes' => null`) sudah dihilangkan di poin 3, maka secara otomatis: **setiap kali surat di-reject lalu di-resubmit ("dikembalikan"), catatan revisi sebelumnya tidak hilang** — tersimpan permanen di baris `document_approvals` yang lama, dan bisa direpetisi berkali-kali (reject → resubmit → reject lagi → resubmit lagi) tanpa kehilangan riwayat manapun, karena setiap siklus membuat baris baru, bukan menimpa baris lama.

Ini sudah dibuktikan saat validasi end-to-end di poin 3 (lihat tabel hasil tinker): catatan *"Format surat salah, tolong revisi nomor surat."* tetap tersimpan di baris `REJECTED` setelah resubmit, tidak menjadi `NULL`.

### Bagian "Ada di Log"

Catatan permintaan ini juga menyebut "ada di log" — perlu klarifikasi apakah maksudnya:
- (a) tersimpan di database (`document_approvals.notes`) — ✅ sudah terpenuhi, atau
- (b) tercatat juga di file log aplikasi (`storage/logs/laravel.log`) lewat `Log::info(...)` — saat ini `WorkflowApprovalEngine` sudah mencatat event `approval_dilakukan` (approve/reject) dan `dokumen_disubmit_ulang` (resubmit) ke log, tapi **tidak menyertakan isi `notes`** di payload log-nya, hanya `document_id`, `approval_id`, `step_order`, dll.

Jika yang dimaksud adalah (b) — catatan revisi juga perlu muncul di log aplikasi — itu belum terpenuhi dan perlu perubahan kecil di pemanggilan `Log::info()`. Belum diimplementasikan, menunggu konfirmasi maksud poin ini.
